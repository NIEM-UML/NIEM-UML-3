Searching

MPD Reference Schemas

From the MPD Specification:

Definition: reference schema – An XML Schema document that meets all of the following criteria:

…

• It is intended to serve as the basis for components in IEPD and EIEM schemas, including subset schemas, constraint schemas, extension schemas, and exchange schemas.

…

[Rule 3-10] A NIEM-conformant IEPD or EIEM MUST contain at least one schema that is either a NIEM reference schema or a subset derived from a NIEM reference schema.

Since it is required that an IEPD or EIEM have at least one schema derived from a NIEM reference schema, and since the number of schema components in the set of all NIEM Reference schemas is quite large, the ability to understand, visualize, navigate, and search NIEM Reference models is an import aspect of NIEM modeling.

Exploring NIEM Reference Models

The entire set of NIEM Reference Models is available to the NIEM-UML modeler.  How many of the NIEM Reference Models are actually viewed in a specific MPD model may vary, depending on how the project was created, how many NIEM models were loaded, and what NIEM models may have been exposed to an IEPD via a base EIEM. 

When NIEM Reference Models are exposed in their original form (as opposed to perhaps EIEM subsets), then the reference models will be viewable in the containment browser starting with the package named “niem”.

 

 

To explore the NIEM Reference Models from the containment browser:

•      Expand the niem package in the containment tree

•      Successively expand nested Package to view

o   Packaging structure.  The Packaging structure reflects the structure of the published NIEM Release.

o   Each  <<InformationModel>> Package represents a schema defined in the published NIEM Release.

o   Classifiers contained by <<InformationModel>> Packages represent one of the following:

•      <<PropertyHolder>> Classes.  These Classes hold properties representing top level elements and/or attributes.

•      <<Choice>> Class.  Represents the contents of an XSD Model Group which is a choice compositor.

•      Otherwise, represents a type within the schema.

o   Properties.  Generally Represents an xsd element or attribute.  Other xsd components represented by Properties include:

•      An XSD Wildcard.

•      The item type of a <<List>> DataType.   The type of the property is the item type of the List.

•      An XSD Model Group with a choice compositor.  The type of the property is a <<Choice>> Class.

•      Each NIEM Reference model may contain one or more diagrams.  Actions on a diagram element within the containment browser include opening the diagram.  The forms of diagram which may be present for each Reference schema include:

o   Inheritance Hierarchy

o   Usage relationships (e.g., <<MetadataApplication>>)

o   Realization relationships (e.g.,<<References>>)

o   Associations between Classifiers

•      For the currently selected element in the containment browser, there are other windows which show specific forms of detail, including:

o   Properties Window.   Shows the properties associated with the selected element.

o   Documentation Window.  Shows documentation/comments associated with the selected element.

Magic Draw Search

 

The MagicDraw search mechanism allows for searching within model elements, symbols, and extensions. Searches may be started via Magic Draw main menu Editèfind (or Ctrl + F).  The types of searches include:

·         Search for model element by name

·         Search for model element by applied stereotype

·         Search for model element by tag and value

·         Search for model element by constraint and value

·         Search for model element by documentation/comment content

 

For each of these searches, elements may be filtered by scope, element type, case sensitivity, and other criteria.

 

NIEM Reference Model Similarity Search

The Magic Draw search mechanism is effective for most modeling scenarios, but has some limitations with respect to discovering Reference Model components:

·         Search is by name pattern.

·         There is no ranking of search results.

·         Search is limited to loaded models.

 

For NIEM models, names of components are constructed according to a pattern defined by ISO 11179 Part 5.  From the NDR:

Names are a simple but incomplete means of providing semantics to data components. Data definitions, structure, and context help to fill the gap left by the limitations of naming. The goals for data component names should be syntactic consistency, semantic precision, and simplicity. In many cases, these goals conflict and it is sometimes necessary to compromise or to allow exceptions to ensure clarity and understanding. To the extent possible, NIEM applies [ISO 11179 Part 5] to construct NIEM data component names.

The set of NIEM data components is a collection of data representations for real-world objects and concepts, along with their associated properties and relationships. Thus, names for these components would consist of the terms (words) for object classes or that describe object classes, their characteristic properties, subparts, and relationships.

[Rule 5-5] (REF, SUB, EXT)

A NIEM component name SHALL be formed by applying the informative guidelines and examples detailed in Annex A of [ISO 11179 Part 5], with exceptions as specified in this document, most notably those specified in Section 9, Naming Rules.

Rationale

The guidelines and examples of [ISO 11179 Part 5] provide a simple, consistent syntax for data names that captures context and thereby imparts a reasonable degree of semantic precision.

NIEM uses the guidelines and examples of [ISO 11179 Part 5] as a baseline for normative naming rules. However, some NIEM components require bending of these rules. Special naming rules for these classes of components are presented and discussed in Section 9. In spite of these exceptions, most NIEM component names can be disassembled into their [ISO 11179 Part 5] constituent words or terms.

 

 

The NIEM Reference Model search is based on a pre-configured search index prepared from all components of all NIEM Reference Models.  The search engine:

·         Enables search of all Reference Models, whether loaded or not.

·         Ranks results for similarity in search criteria. 

·         Determines similarity based on similarity of documentation phrases plus disassembled name.    Component names are disassembled into their [ISO 11179 Part 5] constituent words or terms.  NIEM specific terms such as “Type” are considered noise and excluded from search matching criteria.

·         Search criteria may be based on similarity to locally modeled Classifier, Properties, or EnumerationLiterals.

·         Or search criteria may be a search expression.

 

Supported search expressions are documented at http://lucene.apache.org/core/3_6_0/queryparsersyntax.html

Search expression capabilities include:

o   Wildcards :  te?t, test* , te*t. 

•      To perform a single character wildcard search use the "?" symbol.

•      To perform a multiple character wildcard search use the "*" symbol.

o   Fuzzy similarity: roam~, roam~0.8  Fuzzy searches based on the Levenshtein Distance, or Edit Distance algorithm. To do a fuzzy search use the tilde, "~", symbol at the end of a Single word Term.

•      To search for a term similar in spelling to "roam" use the fuzzy search:

•    roam~

•      This search will find terms like foam and roams.

•      an additional (optional) parameter can specify the required similarity. The value is between 0 and 1, with a value closer to 1 only terms with a higher similarity will be matched. For example:

•    roam~0.8

o   Proximity:    finding words that are within a specific distance away. To do a proximity search use the tilde, "~", symbol at the end of a Phrase. For example to search for a "apache" and "jakarta" within 10 words of each other in a document use the search: 

•       "jakarta apache"~10

o   Relevance: level of matching documents based on the terms found. To boost a term use the caret, "^", symbol with a boost factor (a number) at the end of the term you are searching. The higher the boost factor, the more relevant the term will be.

•      Boosting allows you to control the relevance of a document by boosting its term. For example, if you are searching for

•       jakarta apache

•      and you want the term "jakarta" to be more relevant boost it using the ^ symbol along with the boost factor next to the term. You would type:

•       jakarta^4 apache

o   Boolean:  AND, OR, NOT, +, -.  Terms may be combined through boolean logic operators.

o   Grouping:  use parentheses to group clauses to form sub queries.

 

Using the NIEM Reference Model Search

A search may be started using one of the following procedures:

o   Right click on editable Classifier, Property, EnumerationLiteral

      • Select NIEMèSearch NIEM Reference Models for Similar
      • This will open the NIEM Search Results Dialog with similarity results
    • From Main Menu: Analyzeè Search NIEM Reference Models for Similar
      • This will bring up a dialog for entering search query
      • After entering a query, the NIEM Search Results Dialog will be opened with the query results

 

 

The NIEM Search Results Dialog provides options for facilitating additional searches:

o   Track Tree Selection toggle.  If activated, a new similarity search is performed whenever an element is selected in the containment browser.

o   Track Diagram Selection toggle.  If activated, a new similarity search is performed whenever an element is selected in a Diagram.

o   Text Search.   Opens the search query expression dialog.

 

 

 

The individual columns of the NIEM Search Results Dialog may be resized, dragged to a new position, or used to sort the result table.  The columns include:

o   Rank.  Relative similarity ranking of the reference model element, based on the computed weight.

o   Weight.  The computed similarity weight based on the query criteria.

o   Module.   The library module owning the matched reference model element.

o   Name.  Name of the matched reference model element.

o   Type.  The UML metamodel type or NIEM-UML Stereotype for the matched reference model element.

o   Text.  The phrase fragments which most closely match the query criteria, with highlighting of specific term matches.

Right-clicking a row in the NIEM Search Results Dialog provides a choice of actions which may be performed with respect to the selected NIEM Reference Model element:

o   Select in Containment Tree.  Expands the containment browser tree, if required, and selects the NIEM Reference model element.

o   Select in Inheritance Tree.  Expands the inheritance browser tree, if required, and selects the NIEM Reference model element.

o   Specification.  Opens the Specification dialog for the NIEM Reference model element.

o   Find Similar in NIEM Reference Models.    Searches NIEM Reference Models for similarity with the selected NIEM Reference model element.

o   Add Subset Classifier.  Adds selected NIEM Reference Model Classifier to current MPD model as a subsetted Classifier, including its <<InformationModel>> and other requisite subset elements as required.

o   Add Subset Property.  Adds selected NIEM Reference Model Property to current MPD model as a subsetted Property, including its owning Classifier, <<InformationModel>> and other requisite subset elements as required.

o   Add Subset EnumerationLiteral.   Adds selected NIEM Reference Model EnumerationLiteral to current MPD model as a subsetted EnumerationLiteral, , including its owning Classifier, <<InformationModel>> and other requisite subset elements as required.

o   Load Module.  Loads the NIEM Reference Library which owns the selected NIEM Reference Model element.
