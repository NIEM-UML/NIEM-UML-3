Managing an EIEM

Creating an EIEM

To create a new EIEM Project, the following procedure may be used:

 

•      From Magic Draw main menu: FileèNew Project…

•      Select NIEM Modeling category

•      The following options are available for EIEM Projects:

o   NIEM EIEM-All Project.   This project will include all NIEM Reference Models.

o   NIEM EIEM-Core Project.  This project will include the NIEM-Core Reference Model, plus all of its pre-requisite models.

•      Supply the Project Name, Project location if desired

•      Click OK

 

 

The new Project is similar to an IEPD except:

o   There is no exchange <<InformationModel>>

o   There are no XML Sample Document instances

o   The <<ModelPackageDescription>> Component is configured as an eiem

o   The EIEM Model is shared.  The EIEM Model will serve a role similar to NIEM Reference Models from the perspective of the enterprise IEPD modeler.

o   The NIEM Reference Models, NIEM-UML profiles, NIEM-DSL, and XML Primitive Types library are all shared.  It is expected that the view of NIEM Reference Models is managed by the EIEM.  Initially, the NIEM Reference Models are all directly accessible to an IEPD modeler.  The EIEM model may subset some or all of the NIEM Reference Models in order to hide complexity.  For subsetted NIEM Reference Models, the EIEM may disable sharing of the corresponding original NIEM Reference Models (this is implemented via IEPD Templates).  

 

The New Project includes:

o   NIEM Reference Models

o   NIEM-UML Profiles

o   XML Primitive Types

o   NIEM-DSL

o   <<ModelPackageDescription>> Component, configured as an IEPD and with required <<ModelPackageDescriptionFile>> Usages to changelog, xml samples, business-rules

o   Skeletal exchange, extension, subset models to meet minimum NDR/MPD configuration requirements

o   XML Sample Documents, as required by MPD.  These are expected to be replaced by XML Sample Documents based on the modeled exchange <<InformationModel>>.

o   A change log configured for a new IEPD.

o   Business-rules package to manage business constraints and validation.

o   Various skeletal diagrams with some hints on how to start modeling the IEPD.

 

Reduce Complexity

From the NIEM Business Information Exchange Components document:

Development of an IEPD involves discovering and defining the business data needs; mapping the business data to NIEM components and extension components; defining the extension components; developing NIEM subset schemas (to exclude NIEM content not used in a particular IEPD); developing extension schemas; and developing the exchange schema.

As an organization develops more and more IEPDs, and gains experience in the process, a number of questions inevitably arise:

·         Performing all the steps to develop an IEPD can be time consuming, and developing many IEPDs can be costly. How can an organization realize economies of scale and reduce efforts as more IEPDs are developed?

·         NIEM encourages the reuse of IEPDs as a whole, but in many cases information exchanges, even within the same general business area, are not exactly the same, and reuse of the entire IEPD may not be possible. Is there a better way to reuse artifacts developed for one IEPD in the development of another?

·         NIEM has a great deal of content, most of which may not be applicable to a particular organization’s business and won’t be used by that organization for information sharing. It is often difficult for IEPD developers to find the appropriate NIEM content because there is so much to sift through. Is it possible to search only the NIEM components used by the organization’s business?

·         Many IEPDs have similar business content; but if the IEPDs are developed at different times and/or by different people, similar business content may be reflected in arbitrarily different ways. How can an organization ensure that similar business content is reflected the same way in all IEPDs with that content?

·         Extensions developed for one IEPD may be difficult to discover and reuse in another IEPD. How can organizations define and manage extensions specific to their business needs in a way that facilitates discovery and reuse of such extensions?

…

As an organization develops IEPDs, the organization may realize that many of its IEPDs have similar business content. A collection of closely related business data could be organized at an object level and defined as extension data components. These extension components could be referred to as Business Information Exchange Components (BIEC), because they are either specific to an organization’s business or they represent a more general line of business that crosses organizational lines. Often they are business data components developed and used by multiple organizations within the same community of interest (COI).

The use of BIECs has the potential for simplifying IEPD development and increasing consistency of the business object definitions at all steps in the process, including exchange content modeling, mapping to NIEM, creating NIEM extension components, and generating XML schemas.

…

Once an organization defines a number of BIECs, the collection of BIECs can either be used as a library of business components or can form the basis of an Enterprise Information Exchange Model (EIEM). Instead of developing each IEPD by mapping business data to NIEM and extension components, subsetting NIEM, and defining extensions, organizations can perform much of the IEPD development by mapping high-level business content to existing BIECs in their own EIEM. An information sharing enterprise is a group of organizations with business interactions that agree to exchange information, often using multiple types of information exchanges. The member organizations have similar business definitions for objects used in an information exchange and can agree on common definitions for BIECs and an EIEM.

…

An Enterprise Information Exchange Model (EIEM) is a content data model of data components that meets the specific information sharing business needs of an enterprise. In a sense it is a tailored adaptation of NIEM, subsetted, extended, and constrained to meet those needs. The EIEM consists of:

·         A “master” subset of NIEM that contains all NIEM components used or expected to be used by the enterprise.

·         One or more extension schemas that define BIECs and other extensions used or expected to be used by the enterprise.

 

For an enterprise that creates and maintains an EIEM, its IEPD development process is no longer required to create NIEM subsets or extensive mappings of business content to the XML data model—for the most part, this work has been accomplished in the process of EIEM development. IEPD developers can reuse definitions, the master NIEM subset, and business content mappings encapsulated in the BIECs that comprise the enterprise model.

…

The goals of using BIECs and an EIEM are to reduce the effort required to create a set of IEPDs and the incremental cost of creating a new IEPD and to increase reuse and consistency among IEPDs used by an enterprise to exchange information. Specifically:

·         Improve the consistency of information representation across multiple IEPDs developed by an information sharing enterprise.

·         Define and manage extensions specific to an enterprise’s business needs in a way that the extensions can be discovered and reused.

·         Reduce the efforts required to develop multiple IEPDs by:

o    Defining BIECs once and reusing them in multiple IEPDs.

o    Mapping IEPD business data primarily to NIEM and extensions at the BIEC level instead of lower levels.

·         Make it easier to find NIEM content by limiting searches primarily to the NIEM components used by the enterprise.

 

The use of BIECs and EIEMs can extend the value of NIEM beyond its use in IEPD development.

…

If desired, a subset of the EIEM (a subset of the extension schemas and a subset of the master NIEM subset schemas) can be developed for each IEPD to more specifically constraint the EIEM components used in the IEPD.

 

 

In addition to enterprise-specific extension models, an eiem contains enterprise-specific views (subsets) of NIEM Reference models.   These subsets will be exposed to the IEPD modeler, who may further subset and tailor the EIEM components, consistent with the goals of the enterprise.

 

 

Modularizing the EIEM

An EIEM often includes one or more extension <<InformationModel>>s.  These extension models may be oriented towards specific communities of interest (COI) within the enterprise.  The development and use of such extension models may be largely independent from other COI extension models.   Common concepts across the enterprise are captured in an enterprise-specific “core” model.

To facilitate the parallel development of COI extension models, an EIEM may be modularized.  In this scenario, there may be a “core” enterprise model project plus multiple COI extension model projects.  The COI extension model projects would be tailored adaptations of the “core” enterprise model; subsetted, extended, and constrained to meet the needs of the COI.

 

A COI extension model project may be created using the following procedure:

·         Create an EIEM SubModule Project

o   Magic Draw Main menu FileèNew Project.   This will open the New Project Dialog.

o   In the NIEM Modeling category, select one of the NIEM EIEM-* Projects

o   Complete the New Project Dialog

 

 

·         Use the EIEM Core module:

o   Magic Draw Main menu FileèUse Module.   This will open the Use Module Dialog.

o   On the Select Module Page:  Select the EIEM “Core” model

o   On the Module Setting Page: Select read-only, always load

o   Complete the Use Module Dialog.

·         Be sure to Avoid circular module dependencies

 

·         Model the COI extension <<InformationModel>>

o   Subset  <<InformationModel>>s from core to meet the needs of the COI

o   Add COI extension <<InformationModel>>

§  Extend/restrict components from the eiem core model.

§  Add components as required

·         By default, NIEM-UML Profiles are properly applied and <<ModelPackageDescription>> Component is shared (referenceable) from IEPD.

o   Be sure that NIEM-UML Profiles remain properly applied.

o   Be sure that <<ModelPackageDescription>> Component remains shared so that IEPD may reference it via <<ModelPackageDescriptionRelationship>>.

 

In addition to NIEM-UML models, an EIEM may have other artifacts which are intended for reuse by IEPD modelers.  This may include, for example, enterprise-specific UML Profiles (beyond NIEM-UML) which are to be applied to IEPD models.

The IEPD Template

To use an EIEM in the context of IEPD development, the modeler must create an IEPD Project, use a set of EIEM modules and other requisite artifacts,  subset EIEM <<InformationModel>>s,  create the MPD required <<ModelPackageDescriptionRelationship>> between the IEPD and the EIEM, and develop the exchange-specific <<InformationModel>>s.

 

To facilitate the use of EIEMs within an enterprise, the initial configuration/setup for an IEPD Project may be encapsulated in an IEPD Template.  An IEPD Template is basically a model which can be replicated for use as a starting point for IEPD development.  The IEPD Template should ensure that a synchronized set of EIEM modules and other artifacts are loaded, that <<ModelPackageDescriptionRelationship>> Usages are properly established with the EIEM models, and that common/useful EIEM subset <<InformationModel>>s are provided.

 

Since an EIEM may contain several COI extensions, it may be useful to provide multiple IEPD Templates, each tailored to a COI or other perspective of IEPD usage.

Resource Bundle

A resource bundle may be used to facilitate the management, distribution, and use of EIEMs and their associated artifacts within an enterprise.  A resource bundle is used import all the EIEM artifacts as a unit into the IEPD model developer’s environment.    The resource bundle includes:

·         The EIEM Core model library

·         All COI EIEM extension model libraries.

·         All EIEM requisite Profiles and other artifacts.

·         All IEPD Templates

 

A resource bundle is a zip file constructed from a file folder having a specific structure which parallels the Magic Draw directory structure.  The required resource bundle file structure and content may be produced with the following procedure:

•      Create an eiem resource folder

•      Create “modelLibraries” folder within the resource folder and put  all EIEM models in it.  This will result in the eiem model libraries being loaded to the modelLibraries folder within Magic Draw.

·         Create “templates” folder within the resource folder.

·         Create a subfolder within “templates” for the IEPD Templates.  The name of this subfolder will ultimately be displayed in the “New Project” Dialog.

·          Put the IEPD template models in the subfolder just created.   The name of the IEPD Templates will ultimately be displayed under the subfolder within the “New Project” Dialog.

 

 

The Resource Builder

Magic Draw has a no-cost plugin named “Resource Builder”.  That plugin is used to build the resource bundle zip file based on the resource folder we just constructed, plus a few internal Magic Draw files used to manage the install/uninstall resource process.  The following procedure may be used to obtain the “Resource Builder”:

•      Start Resource/Plugin Manager

o   From Magic Draw main menu:  HelpèResource/Plugin Manager.   This will open the Resource/Plugin Manager dialog.

•      From the Resource/Plugin Manager Dialog:

o   Set Filter:  All Resources

o   Scroll down to Category: Plugins (no cost)

o   Select Resource Builder checkbox.

o   Push Download/Install.

o   When download is complete, close dialog.

•      Restart Magic Draw

 

The resource builder may be used to build the distributable resource bundle zip file, using the following procedure:

 

•      From the Magic Draw Main menu:  ToolsèBuild Custom Resource.  This will result in opening the Custom Resource Building Wizard.

•      For step 1:  Locate your eiem resource folder

•      Step through and Complete Wizard Dialogs as illustrated below

•      For step 4: the Resource file: is the name of the distributable resource file, the suffix should be .zip

•      Upon completion of the wizard, the distributable *.zip is generated

 

 

 

 

 

Importing the EIEM Resource Bundle

The zipped resource file should be distributed to IEPD model developers.  Each IEPD model developer may load the resource using the following procedure:

 

•      From the Magic Draw Main menu: helpèResource/Plugin Manager.  This will open the Resource/Plugin Manager dialog.

•      From the Resource/Plugin Manager dialog: Click Import.  This will open a file selection dialog.

•      From the file selection dialog:  Select downloaded eiem resource zip

•      Close all dialogs.

•      For this type of distribution, no restart is required.

 

 

Creating and IEPD Project from the IEPD Templates

The IEPD model developer may now create a new IEPD based on the provided IEPD templates using the following procedure:

 

•      From the Magic Draw Main menu: FileèNew Project…   This will open the New Project Dialog.

•      On the left panel of the New Project Dialog, Select the Other Category

•      Within the Other Category, select Project from Template

•      On the right panel of the New Project Dialog:

o   Scroll down and open the eiem-specific IEPD template folder

o   Within the folder, select the applicable IEPD Template

•      Set project name and Project location as appropriate.

•      Push OK

•      Result will be a new Project, as specified in the corresponding IEPD template folder.
