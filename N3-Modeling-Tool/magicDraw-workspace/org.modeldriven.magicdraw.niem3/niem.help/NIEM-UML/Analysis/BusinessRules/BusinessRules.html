MPD Business Rules

An MPD Business Rule is an artifact used to document constraints beyond the capability of NIEM and XML Schema, it may be used to validate or verify that such constraints are satisfied.

 

From the MPD Specification:

An EIEM may also contain business rules … tailored to the enterprise and designed to restrict variability in use of NIEM data components.

…A normative NIEM BIEC Specification will supplement or obviate constraint schemas with consistent and formal techniques for representing business rules within NIEM components.

… In the future, NIEM will prefer business rules over constraint schemas as the primary method for further constraining a base schema set.

… [Rule 4-17] A NIEM IEPD or EIEM master document SHOULD (at a minimum) describe the MPD purpose, scope, business value, exchange information, senders/receivers, interactions, and references to other documentation.

… The following are some common examples of sections in or separate artifacts associated with the master document:

 

·         Business rules

…In addition to documentation artifacts, a variety of other optional files can be added to an MPD to facilitate tool support and make reuse, adaptation, and/or implementation easier…  Examples include … business rules..

MPD Artifact description:

Business Rule. An artifact used to document constraints beyond the capability of NIEM and XML Schema; may be used to validate or verify that such constraints are satisfied..

Our orientation in modeling MPD Business Rules is to ensure an executable validation/verification can be implemented and that it can be used to validate/verify modeled XML Instance documents.

 

 

UML Constraints

Within a UML model, business rules are represented as Constraints.   From the UML Specification for Constraint:

A constraint is a condition or restriction expressed in natural language text or in a machine readable language for the purpose of declaring some of the semantics of an element.

Constraint contains a ValueSpecification that specifies additional semantics for one or more elements. Certain kinds of constraints are predefined in UML, others may be user-defined. A user-defined Constraint is described using a specified language, whose syntax and interpretation is a tool responsibility. One predefined language for writing constraints is OCL. In some situations, a programming language such as Java may be appropriate for expressing a constraint. In other situations natural language may be used.

Constraint is a condition (a Boolean expression) that restricts the extension of the associated element beyond what is imposed by the other language constructs applied to the element.

A Constraint represents additional semantic information attached to the constrained elements. A constraint is an assertion that indicates a restriction that must be satisfied by a correct design of the system. The constrained elements are those elements required to evaluate the constraint specification. In addition, the context of the Constraint may be accessed, and may be used as the namespace for interpreting names used in the specification. For example, in OCL ‘self’ is used to refer to the context element.

Constraints are often expressed as a text string in some language. If a formal language such as OCL is used, then tools may be able to verify some aspects of the constraints.

In general there are many possible kinds of owners for a Constraint. The only restriction is that the owning element must have access to the constrainedElements.

The owner of the Constraint will determine when the constraint specification is evaluated..

 

Use of OCL Expression in UML Models

The definition of Constraint within UML is fairly general.   The specific use and interpretation of OCL based Constraints within UML Models is detailed by the OCL Specification:

For every occurrence of an OCL expression three things need to be separated: the placement, the contextual classifier, and

the self instance of an OCL expression.

• The placement is the position where the OCL expression is used in the UML model (e.g., as invariant connected to class Person).

• The contextual classifier defines the namespace in which the expression is evaluated. For example, the contextual classifier of a precondition is the classifier that is the owner of the operation for which the precondition is defined.  Visible within the precondition are all model elements that are visible in the contextual classifier.

• The self instance is the reference to the object that evaluates the expression. It is always an instance of the contextual classifier. Note that evaluation of an OCL expression may result in a different value for every instance of the contextual classifier.

 

…An invariant constraint is a constraint that is linked to a Classifier. The purpose of an invariant constraint is to specify invariants for the Classifier. An invariant constraint consists of an OCL expression of type Boolean. The expression must be true for each instance of the classifier at any moment in time.

 

We use the invariant constraint mechanism to represent business rules, they are always specified in the context of a Classifier.

 

InstanceSpecification

We use the UML InstanceSpecification to represent an instance of a Classifier, and thus provide a tangible representation for instance in an OCL Expression evaluation.

From the UML Specification:

An instance specification is a model element that represents an instance in a modeled system.

 

An InstanceSpecification in NIEM-UML will be subject to the following kinds of validation:

·         UML Constraints

·         XSD Constraints, including validation of value specifications with respect to defined facets

·         NDR and MPD rules regarding XML Instances

·         OCL Constraints defined on corresponding Classifier

Operations

The OCL Specification includes a limited set of Operations for various types.   For the NIEM-UML environment, this set of Operations is extended to include:

·         Many of the side-effect-free additional OCL Operations defined by QVT.   This includes many of the additional Operations defined on Objects, Elements, Lists, Strings, Numeric Types.

·         Operations on Strings (i.e., the UML PrimitiveType String)  have been replicated for use with XMLPrimitiveTypes::string.

·         Many convenience accessors defined with respect to eclipse UML have been added to the OCL.

·         In the future, there may be additional Operations (e.g., from XPath library), depending upon the NIEM technical direction for constraint specification.

Creating the Validation Suite

When the model is created via template, then there will be a default Validation Suite within the model, properly identified as such via a <<ModelPackageDescriptionFile>> Usage from the <<ModelPackageDescription>> Component to an appropriately stereotyped <<activeValidationSuite>> Package within the Model.

 

To create a Validation Suite from scratch:

·         Create an <<activeValidationSuite>>:

o   From the Magic Draw Main menu:  AnalyzeèValidationèValidation Rules

o   (Example Validation Options Dialog shown on previous figure)

o   Create New Validation Suite (icon push button, upper left)

o   Complete dialog sequence for creation of an <<activeValidationSuite>> Package.

·         Identify the new Suite as belonging to the <<ModulePackageDescription>> Component:

o   Add a <<ModelPackageDescriptionFile>> Usage from the the <<ModulePackageDescription>> Component to the <<activeValidationSuite>>;

o   Set the <<ModelPackageDescriptionFile>>  tags appropriately:

§  purposeCode=business-rules

§  natureCode=xmi

 

 

Provisioning Schematron

Schematron is a rule-based schema language defined by an ISO Specification.  It contains assertions concerning a set of documents.   The assertions are specified using a query language.  There is, for each Schematron Schema, a query language binding to a concrete language.  We bind provisioned Schematron Schemas to xpath2. 

 

A Schematron Schema is an XML document which organizes assertions and provides a context for assertion evaluation.  The basic containment structure is SchemaèPatternèRuleèAssert.  We map a NIEM-UML model such that Schema corresponds to an MPD, Pattern is an <<InformationModel>> (which maps to an xml schema), Rule is a Classifier (which maps to an xml type), and Assert is a Constraint.  A Schematron Rule establishes context for its contained Assertion, just as a UML Classifier establishes context for its contained Constraints.  A Schema Assertion uses XPATH to express a Boolean value, just as we require a Constraint to use OCL to express a Boolean value.  Although different syntactically, XPATH and OCL provide similar capabilities in terms of navigation, testing, looping, etc.

 

Schematron can be specified in various ways: it may be embedded as annotations within an XML Schema, it may be fragmented as modules which can be included in another Schematron Schema, it may have abstract components.  Any of these forms of expression must be pre-processed: embedded Schematron is extracted from there host schemas, included Schematron modules are assembled into a single document, abstract components are resolved to concrete components.  The result is a single Schematron Schema which serves as the basis for Schematron processing, as shown in the following figure.  We provision the business rules of an IEPD directly to Schematron Schemas. 

 

A Schematron Schema may be processed directly by various Schematron engines.  Alternatively, the Schematron Schema may be transformed, via XSLT, into a Validating XSLT stylesheet.  In addition to provisioning the Schematron Schema, we transform the Schematron Schema into a Validating XSLT stylesheet.  The Validating XSLT stylesheet may then be executed directly against XML Instance documents to produce a Validation Report, an XML document which records the validation steps and flags violations.  The provisioned Validating XSLT stylesheet must be executed with a schema-aware XSLT 2 processor.  Since such processors are commercial products, and consequently not included in the modeling environment, we are currently not provisioning the Validation Reports for provisioned XML Sample Instance documents.

 

The provisioning mechanisms used include:

·         QVT to generate Schematron Schema from the NIEM-UML model.  This is based on our eclipse emf model for Schematron and our model for OCL with Magic Draw UML Binding.

·         XSLT (Saxon) to transform Schematron Schema to Validating XSLT stylesheet, based on an open source XSLT which implements that mapping.

It should be noted that the context for applying rules is based on the type, not the element.  This will generally make it easier to express business rules, aligns with the OCL context establishment, and avoids some complicated expressions based on element oriented criteria.  Most OCL constructs, other than reflective operations, are mappable to XPATH.
