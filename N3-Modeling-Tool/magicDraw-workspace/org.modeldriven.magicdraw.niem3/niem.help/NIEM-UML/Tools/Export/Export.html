Model Package Description

 

From the NDR:

[Definition: NIEM-conformant schema]

An XML Schema document is a NIEM-conformant schema if and only if it is a reference schema, a subset schema, an extension schema, an exchange schema, or a constraint schema.

 

From the Model Package Description Specification:

 

Definition: Model Package Description (MPD) – A set of related W3C XML Schema documents and other supporting files organized as one of the five classes of NIEM schema sets:

•          Release (major, minor, or micro).

•          Domain update (to a release).

•          Core update (to a release).

•          Information Exchange Package Documentation (IEPD).

•          Enterprise Information Exchange Model (EIEM).

 

Definition: artifact – A single file with a defined purpose or a set of files logically grouped for a defined purpose. An MPD is a collection of related artifacts.

[Rule 4-1] An MPD MUST contain an XML mpd-catalog artifact that validates with the NIEM MPD catalog schema (XSD) and that resides in the root directory of the MPD and bears the file name "mpd-catalog.xml".

[Rule 4-12] Every MPD that is a reference schema set (i.e., NIEM releases, core updates, and domain updates) MUST contain an XML change log artifact..

[Rule 4-13] Every MPD that is an IEPD or EIEM MUST contain a change log…

A Master Document is required for each IEPD and each EIEM.  The MPD Specification requires the Master Document to include the following types of information:

 

•      Purpose of this MPD.

•      Scope of its deployment, usage, and information content.

•      Business value and rationale for developing it.

•      Type of information it is intended to exchange (in business terms).

•      Identification of senders and receivers (or the types of senders and receivers).

•      Typical interactions between senders, receivers, and systems.

•      References to other documentation within the MPD, and links to external documents that may be needed to understand and implement it.

The MPD does not include a concrete specification for the structure of a Master Document.   The structure of a Master Document is often defined at an enterprise or other organizational level.  One commonly adopted set of guidelines for IEPD documentation is:

MPD and NIEM-UML

There exists an isomorphic relationship between NIEM-compliant artifacts and a NIEM-UML Model.  The relationship is formalized as an OMG QVT Model To Model Transformation.  The QVT transforms between instances of a NIEM-UML Profiled UML Model and instances of other MOF Models. 

The meta-model for MPD Schema artifacts is implemented using the eclipse ecore-based XSD model.  The facility provides for serializing the model as an XML Schema in its native format.  Thus, an MPD Schema may be viewed as both an instance of XMLSchema.xsd (as defined by w3c) and an instance of XSDSchema.ecore.

The MPD Catalog, Changelog, and wantlist are all defined in terms of XML Schema.  An eclipse facility has been used to generate an *.ecore implementation for each of the schemas.  The ecore implementations serialize model instances as xml documents.  Thus, the MPD Catalog is both an instance of the mpd-catalog-1.0.xsd Schema and an instance of mpdcatalog.ecore (similarly for change log and wantlist).

There are actually two QVT transformation sets: one transforms from UML to MPD artifacts, another transforms from MPD artifacts to UML.  The round-trip transformations should be lossless for NIEM-conformant artifacts.

In those cases where MOF based models are not as readily available, transformations from UML to MPD artifact use OMG Model To Text transformations.  This form of transformation is used to provision:

•         XML Instance Documents from the UML model for Instance Documents.

•         The Master Document

o   This transformation uses both the Catalog provisioned by QVT and the UML Model, including its <<References>> to NIEM Reference Models.

Additional MPD artifacts are provisioned by replicating static resources into the target MPD File Structure.  This includes replication of NIEM Infrastructure Schemas into the target MPD structure.

 

 

Provisioning an MPD from The Model

An MPD with all MPD-required artifacts is provisioned from a NIEM-UML Model using the following procedure:

•      From the Magic Draw Main Menu: FileèExport ToèNIEM MPD

 

 

Assuming the NIEM-UML model validates, The resulting MPD includes all required artifacts:

•      Schemas. 

o   An XML Schema is generated for each <<InformationModel>> in the transitive closure of <<InformationModel>>s referenced directly or indirectly by the <<ModelPackageDescription>> Component.

o   The NIEM Infrastructure Schemas are replicated into the target MPD file structure.

•      Catalog.  The core portion of the Catalog is specified by the <<ModelPackageDescription>> Component and its modeled relationships with various MPD artifacts.   The catalog is additionally populated by derived/inferred relationships, such as the transitive closure of <<InformationModel>>s, the MPD definition of file-sets and folders, artifacts which are automatically provisioned such as the master document and catalog, etc.

•      Changelog.  This is provisioned based on the modeled change log, which is generated using model difference analysis.  Detail component metadata is extracted from the current model as well as the referenced base-line model.

  • Master-Document (IEPD, EIEM).    This is provisioned based on the UML Models, Catalog, and perhaps other artifacts in the environment.  The structure of the document is based on GJXDM Information Exchange Package Documentation Guidelines.

•      XML Instance Documents (IEPD).   Instance documents are provisioned based on the UML Representation of Instance Documents.

•      +Constraint Schemas.   Constraint schemas are used in the model to represent overrides to the base set of schemas.   The provisioned set of constraint schemas is essentially a copy of the base set of schemas, with modeled overrides applied.

•      +Wantlist.   Based on subsets of NIEM Reference Models.

•      +Model.  A replication of the UML model into the target MPD structure.

 

The Directory Structure provisioned into the target MPD is based on the following factors:

•      The MPD Catalog, Changelog, Master-Document are always provisioned to root directory, per MPD requirement.

•      If there is a <<ModelPackageDescriptionFile>> with an explicit relativePathName, then the relativePathName is used as the target location of the artifact.

•      The location for all other cases of artifact, including Schemas, Constraint Schemas, XML Instances, etc. are based on the UML Qualified Name of the artifact.   More specifically, it is the relative path to the artifact from the nearest package which has NIEM-UML Profiles applied.

 

Provisioning Options

Some of the behavior for provisioning an MPD from the model may be altered by a set of Provisioning Options, which includes options to:

•         Merge PSM.   The provisioning process is done in two stages:  transformation from PIM to PSM perspective; transformation from PSM to MPD artifact.  This option controls whether or not the intermediate PSM model is preserved.   Performance is generally better when the intermediate PSM model is preserved.

•         Provision Master Document.  This option controls whether or not the master document is provisioned.  Performance of the overall transformation is improved if the Master Document is not provisioned.

•         Provision Master Document Diagrams.  This option controls whether or not diagrams are provisioned into the master document.  Performance of master document generation is improved when diagrams are not provisioned.

•         MPD Export Working Directory.  The MPD root directory(s) and any working directories used by the MPD provisioning process will be contained by this working directory.  This may be an absolute file location, location relative to the MD Install directory, or may be composed from path variables such as <project.dir>.

 

The MPD Provisioning Options may be viewed/edited using the following procedure:

 

•      From the Magic Draw Main menu: OptionsèProject.  This will open the Project Options dialog.

•      In the Project Options Dialog, Select “General Project Options” in left pane.

•      In the right pane for “General project options”, scroll to the NIEM-UML category.

•      Click on Property;   Description appears in right bottom pane.

•      Click on Value to edit the value.
