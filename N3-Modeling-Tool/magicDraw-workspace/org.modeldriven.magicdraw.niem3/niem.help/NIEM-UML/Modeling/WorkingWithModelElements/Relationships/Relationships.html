Relationships

A UML Relationship is an abstract concept that specifies some kind of relationship between elements.  The iconic representation for relationships is some variant of a diagonal line from the lower left of the relation icon to the upper right of the relation icon.  The diagonal line may have adornments which provide a visual cue as to the underlying Relationship specialization.  When drawing relationships, the first selected element corresponds to the from and the target element corresponds to the to.

A directed relationship represents a relationship between a collection of source model elements and a collection of target model elements.  A directed relationship is depicted with an arrow head on one end of the diagonal line.  The end which has the arrow head represents the supplier for a directed relationship; the other end represents the client.

Although not derived from a UML DirectedRelationship, Associations are logically directed relationships when a member end is not navigable.  The iconic representations for directed associations are also depicted with an arrow head on one end of the diagonal line.

Creating Relationships from the NIEM Tool Palette

Relationships within the NIEM Tool Palette are organized into categories:

•      Derivation.   This category generally represents derivation from a base type.  Derivation Relationships include:

o   <<Restriction>>.  The base type is the supplier of the relationship; the derived type is the client.  The semantics and constraints of an xsd restriction apply to the relationship.

o   Generalization.    An un-stereotyped Generalization normally corresponds to an xsd extension.   Exceptions to this interpretation include:

§  Generalization between simple types, in which case the Generalization corresponds to an xsd restriction.

§  Generalization between a non-<<AugmentationType>> and an <<AugmentationType>>.  This represents an Augmentation Element whose type is the <<AugmentationType>>.

o   (Inverse) Generalization.

o   <<RolePlayedBy>> Generalization.   This is an alternative representation for a <<RoleOf>> Property, wherein the supplier represents the type of the implicit Property.

o   <<Augments>> Generalization.   This is an alternative representation for an <<AugmentationApplication>>, wherein the supplier represents the constraint on the application of augmentation elements.

•      Application.  This category represents the abstract NIEM-UML <<Application>> Stereotype, which represents application of the NIEM concept of “appliesTo”.   Application relationships include:

o   <<AugmentationApplication>>.    This relationship is always between a Property whose type is an <<AugmentationType>> Class and a Class. It represents a constraint on a NEIM augmentation property that limits the application of the property to specific types.

o   <<MetadataApplication>>.   This relationship is always between a <<MetadataType>> Class and another Class. It represents a constraint on a NIEM <<metadata type>> that limits the application of the NIEM <<metadata type>> to specific types.

•      Model Package Description.  This category includes the NIEM-UML relationships which represent MPD Catalog references to external artifacts.  Model Package Description relationships include:

o   <<ModelPackageDescriptionFile>>.    Represents a relationship between a <<ModelPackageDescription>> Component or a <<ModelPackageDescriptionFileSet>> Component and an artifact to be included in the MPD.

o   <<ModelPackageDescriptionRelationship>>.   Represents a relationship between <<ModelPackageDescription>> Components or between a <<ModelPackageDescription>> Component and another resource (such as a NIEM specification).

•      UML Un-stereotyped Relationships.  This category includes un-stereotyped UML Relationships which are used in the PIM Perspective of NIEM-UML.  These relationships include:

o   Element Import.  Used as an alternative to a <<ModelPackageDescriptionFile>> to identify an <<InformationModel>> to be explicitly included in the MPD.  The supplier of the import is typically an exchange <<InformationModel>> for an IEPD, it may be one or more extension <<InformationModel>>s for an EIEM.

o   Realization.

o   Usage.   A Usage from a client <<ModelPackageDescription>> Component to a supplier <<ModelPackageDescriptionFileSet>> Component is used to represent inclusion of the <<ModelPackageDescriptionFileSet>> in the MPD represented by the <<ModelPackageDescription>>.  A Usage from a client <<InformationModel>> to a supplier <<InformationModel>> indicates that references from the client <<InformationModel>> to the namespace of the supplier <<InformationModel>> shall be resolved (if possible) as references to the specific schema represented by the supplier <<InformationModel>>.

o   (Inverse) Usage.

•      Associations.  This category includes various forms of Association between Classifiers.  The Association relationships include:

o   Association Class.  An un-stereotyped UML Association Class is an alternative representation for an <<AssociationType>> with two Properties, one for each of the association ends.

o   <<RoleOf>>.   A representation of a NIEM “RoleOf” Element as a directed Association.

o   Reference.  A representation of a NIEM “Reference” Element as a directed non-aggregate Association.

o   Content.  A representation of a NIEM contained Element as a directed aggregate Association.

o   Association.  A representation of a pair of navigable NIEM “Reference” Elements, as an undirected, non-aggregate Association.

•      Other.  This category includes remaining NIEM-UML relationships, including:

o   <<References>>.   This relationship is used in the following ways:

§  A client Property to supplier Property represents the “ref” to a top level element/attribute.  The supplier property may be in the same namespace, or may be in a different namespace.

§  A client <<InformationModel>>, Classifier, or Property to a supplier of the same UML meta-class, in a different <<InformationModel>>, but having the same namespace represents a “subsetting” relationship.

o   <<UnionOf>>.  This relationship is from a client <<Union>> DataType to a supplier DataType.  This relationship represents inclusion of the supplier DataType as a member type of the <<Union>>.

To create a Relationship from the NIEM Tool Palette:

•      Select one of the Relationship tools enumerated above

•      In the diagram, click on the From element

•      Move to, and click on, the To element.   The result of this action is creation of the Relationship.

 

 

 

 

Creating Relationships from Diagram Smart Manipulators

Relationships may be created from the relationship smart manipulators of a selected element on a diagram.  Only the contextually appropriate Smart Manipulators for an element are made available for manipulation.

To create a relationship from a smart manipulator:

•      Select an element to be the From element for a relationship.

•      Select a smart manipulator.

•      Select an element to be the To element for a relationship.

 

Creating Relationships from a Browser

Relationships may be created from a browser using one of the following procedures:

 

•      Select the from element

o   Right-click èNew RelationèOutgoing è*   This will open a Selection Dialog for the to element.

o   Select the to element in Selection Dialog

•      Select the to element

o   Right-click èNew RelationèIncoming è*   This will open a Selection Dialog for the from element.

o   Select the from element in Selection Dialog

 

 

 

 

Creating Relationships from a Specification Dialog

Relationships may be created from a Specification Dialog using the following procedure:

 

•      Open the Specification Dialog for selected element.  The Specification Dialog may be opened from a browser or diagram using the context menu for the selected element.

•      Select the Relations category from the left pane of the Specification Dialog.

•      Do one of the following:

o   Click Create Outgoing… push button.    This will result in opening a Selection Dialog for the to element.  In this mode, the element for this Specification Dialog is the from element.

•      Select the to element in Selection Dialog.

o   Click Create Incoming… push button.    This will result in opening a Selection Dialog for the from element.  In this mode, the element for this Specification Dialog is the to element.

•      Select the from element in Selection Dialog.
