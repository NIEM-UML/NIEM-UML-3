XML Sample Instance Documents

MPD Sample Instances

From the MPD Specification:

Sample XML instances are very valuable artifacts to include with IEPDs. As samples of the actual exchange data, instances can help an IEPD implementer to understand the original intent of the IEPD developer. They can be used by an implementer as data points for validation with the IEPD schemas. And finally, the inclusion of valid, meaningful sample XML instances is an indication of IEPD quality. For these reasons, IEPDs require valid sample XML instances.

[Rule 3-11] A NIEM IEPD MUST contain at least one valid sample XML instance (i.e., IEP) artifact for each exchange schema element that can be the root of a corresponding IEP.

The intent of this rule is not to provide a test for all permutations of XML instances that might be possible given the schema definitions and declarations. As the value propositions explain, its purpose is to ensure IEPD developers have tested their own designs, and to provide implementers with examples they can use for additional understanding and guidance. IEPD developers should strive to include sample XML instances that (1) capture real world business cases of data exchanges, and (2) exercise as many data component definitions and declarations in the schemas as possible. While both of these goals may not be achievable in a single sample XML instance, developers have the option to include multiple sample XML instances; however, only one per intended root element is mandatory.

 

(Recommendation from MPD:)

Maintain all sample XML instances that validate to the schemas in a subdirectory of the MPD root directory with a name that contains the substring "XMLsamples".

 

(Artifact description from MPD):

An XML instance document that validates with corresponding defining schemas; a sample IEP.

 

InstanceSpecification

We use the UML InstanceSpecification to represent an instance of an XML Document.

From the UML Specification:

An instance specification is a model element that represents an instance in a modeled system.

 

An InstanceSpecification in NIEM-UML will be subject to the following kinds of validation:

·         UML Constraints

·         XSD Constraints, including validation of value specifications with respect to defined facets

·         NDR and MPD rules regarding XML Instances

·         OCL Constraints defined on corresponding Classifier

 

XML Sample Instance Browser

The XML Sample Instance Browser provides a tree view of a set of InstanceSpecifications representing an XML Document.

The nodes represent XML Elements/Attributes (and their values for SimpleTypes), within the containment structure defined by the corresponding Classifiers.

Typically the Browser is used in conjunction with the Properties Window to assign values to SimpleTypes.

 

 

 

 

Creating an XML Instance Document

An XML Instance Document, according to the MPD, is based on the root node XML Elements defined by an exchange schema.

To create such an Instance:

•      Select :

o   An exchange <<InformationModel>>

o   Or a Classifier in an exchange <<InformationModel>>

o   Or a Property within an exchange <<InformationModel>>

•      Right-clickèNIEMèadd XML Instance Document

•      This action provides a Selection Dialog enabling selection of one or more Properties corresponding to root node exchange Elements.

•      Based on the selected properties, the following forms of element structures are added to the Model:

o   an Instance Document Package for each selected Property.

o   a <<ModelPackageDescriptionFile>> Usage from the <<ModelPackageDescription>> Comonent to the Instance Document Package.

o   The Instance Document Package is populated with InstanceSpecifications, Slots, and ValueSpecifications to represent a fully populated XML Instance Document corresponding to the selected exchange Property.

 

 

Representing XML Instance Documents

 

An XML Instance Document is represented as a set of InstanceSpecifications, Slots, and ValueSpecifications:

·         An InstanceSpecification represents the instantiation of a Classifier.  The name assigned to the InstanceSpecification represents the NIEM id attribute.  By default, the name is assigned a value (approximately) representing its navigation path within the document structure, similar to the xmi canonical id pattern. 

·         The top exchange element node is contained by some Classifier (typically a <<PropertyHolder>>).  An instance of that Classifier serves as the root InstanceSpecification for the document.  It is the only InstanceSpecification which is not contained by some InstanceValue.

·         An EnumerationLiteral is an InstanceSpecification.   It is the instance of an InstanceValue whose type is an Enumeration.

·         A Slot represents the instantiation of a Property.  The name of the Property is displayed as the element name within the Sample Instance Browser.  An InstanceSpecification owns 0..* Slots.  The implied sequence of Slots corresponds to the sequence of Properties defined by the Classifier and its base Classifiers.

·         A ValueSpecification represents one of the 0..* values of a Slot.  A ValueSpecification is either an InstanceValue or a LiteralSpecification.  A ValueSpecification is typed, the type must be consistent with the type of the definingFeature of the owning Slot.

·         An InstanceValue is a ValueSpecification that identifies an instance.  An InstanceValue will be used to represent the values of either an Enumeration or types which are complex types with complex content.  For InstanceValues of an aggregate Property, the instance is (logically) a containment relationship.  For InstanceValues of a non-aggregate Property, the instance represents a “ref” to an InstanceSpecificaiton within the same document.

·         A LiteralSpecification identifies a literal constant which represents the value of SimpleTypes, or types derived from SimpleTypes (other than Enumerations).  The value must conform to the value space defined by the corresponding type, which is constrained by facets of the underlying XML Primitive Type or further restricted by facets applied to the type or any base type.

·         LiteralBoolean, LiteralInteger, LiteralUnlimitedNatural, LiteralReal are used to express literal constants for types based on their XML Primitive Type counterparts.

·         LiteralString is used to express literal constants for all other types.

 

 

Viewing XML Instance Documents

XML Instance Documents are viewed using the NIEM XML Instances Browser, which is by default one of the tabs on the Browser Window.  If it is not displayed, then it can be viewed by the action at Magic Draw main menu: WindowèNIEM XML Instances.

The NIEM XML Instances browser is often used in conjunction with the Properties window, to edit values of LiteralSpecifications.  If the Properties window is not displayed, then it can be viewed by the action at Magic Draw main menu: WindowèProperties.

 

 

 

•      The nodes presented in the browser tree

o   Represent the logical containment structure of an XML Document, in terms of XML Elements.   (As opposed to the physical UML representation, which is a flattened set of Instance Specifications).

o   Represent a combination of underlying UML Elements:

•      Slot,

•      Value Specification,

•      Instance Specification, when the Value Specification is an Instance Value.

o   Display the name of the Slot’s defining Feature (corresponding to an XML Element).

o   Display a value when the Value Specification is a Literal Specification or Enumeration Literal.

o   Display an id=”*” if the node is referenced (as opposed to contained) by another node.

o   Display a ref=”*” if the node references (as opposed to contains) another node.

o   Display a metadata=”*” if the node references metadata defined in another node of the same document.

o   Display a validation icon if the underlying UML Elements have been flagged with a validation annotation.

o   Display a Slot icon if there has been no value assigned to the Slot.

•      Primitive values may be edited using the following procedure: 

o   Select element node. This will result in the Property Window displaying the Properties of the Value Specification.

o   The value may be edited in the “value” Property of the Property Window.

•      Value Specifications which are typed by an Enumeration may be edited using the following procedure:

o   Select element node.

o   Right clickèNIEMèset EnumerationLiteral

o   A Selection Dialog will offer a list of permitted Enumeration Literals.

o   The selected Enumeration Literal will become the instance of the underlying Instance Value.

•      Slots whose defining Feature is not aggregate correspond to NIEM References.  The “ref” may be edited using the following procedure:

o   Select element node.

o   right-clickèNIEMèset Reference Value;

o   A Selection Dialog will offer a list of permitted Instance Specifications within the document.  Permitted Instance Specifications are those whose Classifier conforms to the type of the Value Specification.

o   The selected Instance Specification will become the instance of the underlying Instance Value.

•      The name of an Instance Specification corresponds to a NIEM id.  The id may be edited using the following procedure : 

o   Select element node.

o   right-clickèNIEMèopen InstanceSpecification.  This will open the Specification Dialog for the instance of the underlying Instance Value.

o   Within the Specification Dialog, set the “name” Property to the desired NIEM id.

•      If an element node has not been populated with all Properties defined for its type, then a new element may be added using the following procedure:

o   Select element node.

o   right-clickèNIEMèadd Instance Property;

o   A Selection Dialog will offer a list of permitted Properties within the context of the owning Instance Specification.  The currently active Properties are pre-selected.  The list of Properties may be modified by removing or adding Properties from the selection list. 

o   The result of selection:

•      Properties removed from selection will have their corresponding Slots removed from the document.

•      Properties added to the selection will have a new Slot, and a pre-populated default containment tree will be added to the document.

•      Properties retained within the selection will be unaltered.

•      If an element node has fewer values then its permitted maximum, new values may be added for the element using the following procedure:

o   Select element node.

o   right-clickèNIEMèadd Value;

o   The result is a new Value Specification added to the value of the Slot.  The Value Specification will include a pre-populated containment tree.  The new value will appear in the browser as an additional element node (with the same element name as the originally selected element node).
