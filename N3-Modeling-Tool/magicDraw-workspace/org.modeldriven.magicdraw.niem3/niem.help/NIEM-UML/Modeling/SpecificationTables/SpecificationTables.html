Specification Tables

 

Model Element Specification Dialog

The primary mechanism to view and edit model element detail is via the Magic Draw Specification Dialog.  The Specification Dialog may be opened from many contexts, including:

    • Context Menu for browsers, diagrams: right-clickèSpecification
    • Context Menu for many dialog controls:right-clickèSpecification

 

 

The Specification Dialog:

·         Is organized into a number of categories, as shown on the tree in the left pane. 

·         Each category has a page containing information relevant to the category.  In many cases, the information for a category is presented as a table of Properties.

o   A description of the currently selected category is provided in the upper pane of the Dialog.

o   NIEM-UML Stereotypes relevant to the model element have categories.

·         A Property:

o   May appear within multiple categories.

o   May be presented as a name/value pair.

o   May be editable.  A type- and multiplicity-specific editor enables editing of the value.

o   May be derived, which generally makes the value not-editable.

o   May have actions, such as opening the Specification Dialog for a value of the Property.

o   A description for the currently selected Property is generally displayed at the bottom of the category page.

·         In many cases there will be one or more NIEM specific categories:

o   Properties deemed most relevant to the NIEM category are included on the page.

o   Properties may include Stereotype-specific tags, the most relevant UML Properties, and NIEM derived properties.

o   NIEM derived properties generally represent NIEM concepts which do not have an isomorphic representation in UML.   Derived properties may also be convenience accessors, such as inverse navigation across directed relationships.  Some examples of NIEM derived types:

§  NIEMName.  Computed from the given name (or <<ReferenceName>>) and coerced to a NDR conformant name based on type, context, or other criteria.

§  NIEM Typed Element of Type.  The set of Properties whose type is the current Classifier Specification.

§  Subset NIEM Reference Type.  The Reference model type for which the current Classifier Specification is a subset.

§  NIEM Substitution Groups.  The set of Properties which are substitutable for the current Property Specification.

§  NIEM Base Type.   NIEM Base Type is at most one Classifier derived from either a Generalization or a <<Restriction>>, but not if the Generalization is a <<RolePlayedBy>> or is a Generalization from a non-<<AugmentationType>> to an <<AugmentationType>>. 

 

 

Tables

The Specification Dialog provides all the detail required for understanding a model element.  However, it may sometimes be tedious to single step through Specification Dialogs in order to edit the properties of elements individually, or to track down anomalies in the model.

 

A table:

·         Enables view/edit of multiple elements in a tabular form.

·         Contains a single row per model element.  Properties are columns.  Values are viewed/edited in the boxes of the table.

·         Any property appearing in a Specification Dialog may be used in a table.

·         Columns may be resized, moved, added, removed, and used for sorting.

·         Rows may be easily added to the table:

o   Via drag and drop from Containment Browser

o   Via selection dialogs

 

 

There are several pre-configured tables which may be useful for NIEM modeling.   The action to create these tables is generally to select an element in the Containment tree, right-clickèNIEMètable action.   The result is a table consisting of all sub-objects of the selected element which are of an appropriate type.   The table actions include:

·         Comment table.   Rows of the table are comments from Classifiers, Properties.  Any Classifier, Property in the containment structure which does not already have a comment will have a comment added.  Result is a table which enables editing of comment bodies across all these elements.

·         Properties table.  Rows of the table are Properties.  Result is a table which enables editing of Property attributes such as aggregation, cardinality, name.

·         Types table.  Rows of the table are Classifiers.  Result is a table which may be used to analyze relationships, usages, subsetting, and transitively referenced <<InformationModel>>s of Classifiers,.

·         Type/Properties using another <<InformationModel>>.    Rows of the table are Classifiers and Properties which have a relationship to external <<InformationModel>>s.  Result is a table which may be used to understand where dependencies occur between <<InformationModel>>s.  An incoherent model is one in which there exists more than one <<InformationModel>> with the same target Namespace within the context of an exchange <<InformationModel>>.  This may occur when manually changing a subset model (instead of using the NIEM subsetting actions) or inadvertently cross-wiring between request and response models within the same MPD.  The cause of the incoherence can sometimes be difficult to track down.  Use of this pre-configured NIEM table may assist in tracking down the cause of incoherency.

·         Type/Properties used by another <<InformationModel>>.  Similar to previous table, but used to understand where types/properties are used by other <<InformationModel>>s.
